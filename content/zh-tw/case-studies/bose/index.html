挑戰

作爲一家以高品質音頻設備聞名的公司，[Bose](https://www.bose.com/en_us/index.html) 已經提供了超過五年的聯網產品，隨着這一需求的增長，基礎設施不得不改變以支持它。 “我們需要爲開發者提供一種機制，能夠快速原型化並部署服務一直到生產環境”，首席雲工程師 Josh West 說道。 2016年，公司決定開始從頭構建一個平臺。其主要目標是：“要領先於各個產品組一到兩步，這樣我們就永遠不會匆忙追趕他們的規模”， 雲架構經理 Dylan O'Mahony 說道。

解決方案

從一開始，團隊就知道他們想要一個微服務架構。在評估並原型化了幾種編排解決方案後， 團隊決定爲其在 AWS 上運行的規模化 IoT 平臺即服務（Platform-as-a-Service）採用 [Kubernetes](https://kubernetes.io/)。該平臺還集成了 Prometheus 監控，並於2017年投入生產， 一開始就服務於超過300萬的聯網產品。自那時以來，Bose 已經採用了多項其他 CNCF 技術，包括 [Fluentd](https://www.fluentd.org/)、[CoreDNS](https://coredns.io/)、[Jaeger](https://www.jaegertracing.io/) 和 [OpenTracing](https://opentracing.io/)。

影響

隨着大約 100 名工程師的加入，該平臺現在每年支持跨數十個微服務的 30,000 次非生產部署。 2018 年，有超過 1250 次生產部署。僅一個生產叢集就包含 1,800 個命名空間和 340 個工作節點。 “我們有一個全新的服務，從概念、編碼、部署一直到生產，包括強化、安全測試等，整個過程不到兩個半星期”，O'Mahony 說道。

“我們需要爲開發者提供一種機制，能夠快速原型化並部署服務一直到生產環境”，首席雲工程師 Josh West 說道， “我們希望提供許多雲能力來支持我們的音頻設備和體驗。”

2016年，公司決定開始從頭構建一個物聯網平臺。其主要目標是：“要領先於各個產品組一到兩步， 這樣我們就永遠不會匆忙追趕他們的規模”，雲架構經理 Dylan O'Mahony 說道， “如果他們發佈了一款新的聯網產品，我們希望已經遠遠領先於能夠處理他們可能帶來的任何規模。”

從一開始，團隊就知道他們想要一個微服務架構和平臺即服務。在評估並原型化了包括 Mesos 和 Docker Swarm 在內的編排解決方案後，團隊決定爲其運行在 AWS 上的平臺採用 [Kubernetes](https://kubernetes.io/)。 當時 Kubernetes 仍處於1.5版本，但該技術已經能夠滿足團隊當前和未來大部分的需求。對於 West 來說， 這意味着儲存和網路得到了處理。O'Mahony 強調了 Kubernetes 的可移植性，以防 Bose 決定採用多雲策略。

“Bose 是一家着眼於長遠的公司，”West 說道， “採用一個快速的現成商業解決方案可能在當時適用，但它無法支持我們向前發展， 而這是我們從 Kubernetes 和 CNCF 得到的。”

{{< case-studies/quote image="/images/case-studies/bose/banner3.jpg" author="Dylan O'Mahony, Cloud Architecture Manager, Bose" >}} “團隊中的每個人都思考如何實現自動化，優化流程，儘可能快速地完成任務。當你退後一步， 思考這對於一家擁有50多年曆史的音箱公司來說意味着什麼時，這真的非常不可思議， 我認爲我們使用的工具和與之建立的基礎是其中非常重要的一部分。” {{< /case-studies/quote >}}

團隊花費時間選擇工具，以使開發者的體驗更加輕鬆。“我們的開發者與由我們的運維團隊提供的工具進行交互， 而運維團隊在 Kubernetes 上運行他們所有的工具”，O'Mahony 說道，“我們儘量避免將直接的 Kubernetes 訪問作爲唯一途徑。實際上，理想情況下，我們的開發者甚至不需要知道他們在 Kubernetes 上運行。”

該平臺從一開始就集成了 [Prometheus](https://prometheus.io/) 監控，並於2017年通過後門方式進入生產環境， 一開始就服務於超過300萬的聯網產品。“即使我們爲設計這個平臺的音箱和產品離正式發佈還有一段時間， 但我們確實已經在市場上有一些聯網音箱”，O'Mahony 說道，“我們基本上開始將這些音箱的某些功能及其配套應用指向這個平臺。”

如今，僅 Bose 的一個生產叢集就包含 1800 個命名空間/獨立服務和 340 個節點。 隨着現在約有 100 名工程師加入，該平臺基礎設施現在每年支持跨數十個微服務的 30,000 次非生產部署。2018 年，進行了超過 1250 次生產部署。與 Bose 之前支持遠少次數的部署和服務的部署流程相比， 這是一個驚人的改進。

{{< case-studies/quote image="/images/case-studies/bose/banner4.jpg" author="Josh West, Lead Cloud Engineer, Bose" >}} “CNCF 景觀圖簡明地解釋了從儲存到雲提供商再到自動化等各個不同領域正在發生的事情。 這是我們構建雲基礎設施的購物車。我們可以從不同的過道中進行選擇。” {{< /case-studies/quote >}}

“我們有一個全新的服務，從概念、編碼、部署一直到生產，包括強化、安全測試等，在不到兩個半周的時間內完成部署”， O'Mahony 說道。“每個人都思考如何實現自動化，優化流程，儘可能快速地完成任務。當你退後一步， 思考這對於一家擁有50多年曆史的音箱公司來說意味着什麼時，這真的非常不可思議， 我認爲我們使用的工具和建立的基礎是其中非常重要的一部分。”

許多這樣的技術，如 [Fluentd](https://www.fluentd.org/)、[CoreDNS](https://coredns.io/)、 [Jaeger](https://www.jaegertracing.io/) 和 [OpenTracing](https://opentracing.io/)， 均來自 [CNCF 全景圖](https://landscape.cncf.io/)，West 和 O'Mahony 在 Bose 的雲原生旅程中一直依賴於此。 “CNCF 全景圖簡明地解釋了從儲存到雲提供商再到自動化等各個不同領域正在發生的事情”，West 說道， “這是我們構建雲基礎設施的購物車。我們可以從不同的過道中進行選擇。”

他補充道，“如果不是因爲 Kubernetes 和其他 CNCF 項目的免費開源軟體及其強大的社區支持， 我們永遠不會達到這樣的規模，甚至無法按計劃推出。”

轉向雲原生的另一個好處是：“我們甚至吸引了更多的人才加入 Bose，因爲我們深入參與到 [CNCF 全景圖](http://careers.bose.com)中”，West 說道（是的，他們在招聘）， “這使得許多人能夠做許多偉大的事情，真正將 Bose 帶入了雲的未來。”

{{< case-studies/quote author="Dylan O'Mahony, Cloud Architecture Manager, Bose" >}} “我們正在進行很多工作，以支持 Bose 更多的業務部門，而不僅僅是目前的消費電子部門。正是由於雲原生領域的工具和功能，我們才能夠爲所有試圖創造一些非常棒的體驗的開發者和部門提供如此出色的雲平臺。” {{< /case-studies/quote >}}

在即將到來的一年裏，團隊希望在服務網格和無伺服器方面進行工作，並在全球範圍內擴展。 “通過多區域部署來降低延遲將是我們的一個重點”，O'Mahony 說道，“爲了確保我們在日本、 澳大利亞及其他地區的客戶有良好的體驗，我們希望在更接近他們的地點擁有存在點。 這在 Bose 是前所未有的。”

這不會阻止他們，因爲團隊總是追求宏偉的目標。“我們希望達到數十億的聯網產品” ，West 說道， “我們正在進行很多工作，以支持 Bose 更多的業務部門，而不僅僅是目前的消費電子部門。 正是由於雲原生領域的工具和功能，我們才能夠爲所有試圖創造一些非常棒的體驗的開發者和部門提供如此出色的雲平臺。”

事實上，鑑於平臺已經支持的規模，O'Mahony 認爲，“在這種情況下，不做 Kubernetes 而選擇其他任何東西，我認爲都是愚蠢的。”
