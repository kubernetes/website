挑戰

Nordstrom 希望提高其技術運營的效率和速度，其中包括 Nordstrom.com 電子商務網站。與此同時，Nordstrom 技術公司正在尋找壓縮技術運營成本的方法。

解決方案

在四年前採用 DevOps 轉型並啓動持續集成/部署 (CI/CD)項目後，該公司將部署時間從 3 個月縮短到 30 分鐘。但是他們想在部署環境上走得更快，所以他們開始他們的雲原生之旅，採用與 Kubernetes 協調的 Docker 容器。

影響

爲 Nordstrom 構建 Kubernetes 企業平臺的團隊高級工程師 Dhawal Patel 說，“使用 Kubernetes 的 Nordstrom 技術開發人員現在項目部署得更快，並且能夠只專注於編寫應用程序。”此外，該團隊還提高了運營效率，根據工作負載將 CPU 利用率從 5 倍提高到 12 倍。Patel 說：“我們運行了數千臺虛擬機（VM），但無法有效地使用所有這些資源。藉助 Kubernetes，我們甚至不需要嘗試去提高叢集的效率，就能使運營效率增長 10 倍。”

{{< case-studies/quote author="Dhawal Patel， Nordstrom 高級工程師" >}} “我們一直在尋找通過技術進行優化和提供更多價值的方法。通過 Kubernetes，我們在開發效率和運營效率這兩方面取得了示範性的提升。這是一個雙贏。” {{< /case-studies/quote >}}

當 Dhawal Patel 五年前加入 Nordstrom，擔任該零售商網站的應用程序開發人員時，他意識到有機會幫助加快開發週期。

在早期的 DevOps 時代，Nordstrom 技術仍然遵循傳統的孤島團隊和功能模型。Patel 說：“作爲開發人員，我花在維護環境上的時間比編寫代碼和爲業務增加價值的時間要多。我對此充滿熱情，因此我有機會參與幫助修復它。”

公司也渴望加快步伐，並在 2013 年啓動了首個持續集成/部署（CI/CD）項目。該項目是 Nordstrom 雲原生之旅的第一步。

開發人員和運營團隊成員構建了一個 CI/CD 管道，在內部使用公司的伺服器。團隊選擇了 Chef，並編寫了自動虛擬 IP 創建、伺服器和負載均衡的指導手冊。Patel 說：“項目完成後，部署時間從 3 個月減少到 30 分鐘。我們仍有開發、測試、暫存、然後生產等多個環境需要重新部署。之後，每個運行 Chef 說明書的環境部署都只花 30 分鐘。在那個時候，這是一個巨大的成就。”

但是，新環境仍然需要很長時間才能出現，因此下一步是在雲中工作。如今，Nordstrom Technology 已經構建了一個企業平臺，允許公司的 1500 名開發人員在雲中部署以 Docker 容器身份運行的應用程序，這些應用程序由 Kubernetes 進行編排。

{{< case-studies/quote image="/images/case-studies/nordstrom/banner3.jpg" >}} “瞭解到早期的社區支持和項目迭代指標，我們肯定 Kubernetes 一定會成功的，因此我們以 Kubernetes 爲核心重建了我們的系統。” {{< /case-studies/quote >}}

Patel 說：“雲提供了對資源的更快訪問，因爲我們在內部需要花數週時間才能部署一個虛擬機（VM）來提供服務。但現在我們可以做同樣的事情，只需五分鐘。”

Nordstrom 首次嘗試在叢集上調度容器，是基於 CoreOS fleet 的原生系統。他們開始使用該系統做一些概念驗證項目，直到 Kubernetes 1.0 發佈時纔將正式項目遷移到裏面。Nordstrom 的 Kubernetes 團隊經理 Marius Grigoriu 表示：“瞭解到早期的社區支持和項目迭代指標，我們肯定 Kubernetes 一定會成功的，因此我們以 Kubernetes 爲核心重建了我們的系統。”

雖然 Kubernetes 通常被視爲微服務的平臺，但在 Nordstrom 擔任關鍵生產角色的 Kubernetes 上推出的第一個應用程序是 Jira。Patel 承認：“這不是我們希望作爲第一個應用程序獲得的理想微服務，但致力於此應用程序的團隊對 Docker 和 Kubernetes 非常熱情，他們希望嘗試一下。他們的應用程序部署在內部運行，並希望將其移動到 Kubernetes。

對於加入的團隊來說，這些好處是立竿見影的。Grigoriu 說：“在我們的 Kubernetes 叢集中運行的團隊喜歡這樣一個事實，即他們擔心的問題更少，他們不需要管理基礎設施或操作系統。早期使用者喜歡 Kubernetes 的聲明特性，讓他們不得不處理的面積減少。

{{< case-studies/quote image="/images/case-studies/nordstrom/banner4.jpg">}} Grigoriu 說：“在我們的 Kubernetes 叢集中運行的團隊喜歡這樣一個事實，即他們擔心的問題更少，他們不需要管理基礎設施或操作系統。早期使用者喜歡 Kubernetes 的聲明特性，讓他們不得不處理的面積減少。” {{< /case-studies/quote >}}

爲了支持這些早期使用者，Patel 的團隊開始發展叢集並構建生產級服務。“我們與 Prometheus 集成了監控功能，並配有 Grafana 前端；我們使用 Fluentd 將日誌推送到 Elasticsearch，從而提供日誌聚合”Patel 說。該團隊還增加了數十個開源組件，包括 CNCF 項目，而且把這些成果都貢獻給了 Kubernetes、Terraform 和 kube2iam。

現在有 60 多個開發團隊在 Nordstrom 上運行 Kubernetes，隨着成功案例的湧現，更多的團隊加入進來。Patel 說：“我們最初的客戶羣，那些願意嘗試這些的客戶羣，現在已經開始向後續使用者宣傳。一個早期使用者擁有 Docker 容器，他不知道如何在生產中運行它。我們和他一起協作，在 15 分鐘內，我們將其部署到生產中。他認爲這是驚人的，他所在的組織更多的人開始加入進來。”

對於 Nordstrom 而言，雲原生極大地提高了開發和運營效率。現在，使用 Kubernetes 的開發人員部署速度更快，可以專注於在其應用程序中構建價值。一個團隊從 25 分鐘的合併開始，通過在雲中啓動虛擬機來進行部署。切換到 Kubernetes 的過程速度是原來 5 倍，將合併時間縮短爲 5 分鐘。

{{< case-studies/quote >}} “藉助 Kubernetes，我們甚至不需要嘗試去提高叢集的效率，目前 CPU 利用率爲 40%，較之前增長了 10 倍。我們正在運行 2600 多個客戶 Pod，如果它們直接進入雲，這些 Pod 將是 2600 多個 VM。我們現在在 40 臺 VM 上運行它們，因此這大大降低了運營開銷。 {{< /case-studies/quote >}}

速度很重要，並且很容易證明，但也許更大的影響在於運營效率。Patel 說：“我們在 AWS 上運行了數千個 VM，它們的總體平均 CPU 利用率約爲 4%。藉助 Kubernetes，我們甚至不需要嘗試去提高叢集的效率，目前 CPU 利用率爲 40%，較之前增長了 10 倍。我們正在運行 2600 多個客戶 Pod，如果它們直接上雲，這些 Pod 將是 2600 多個 VM。我們現在在 40 臺 VM 上運行它們，因此這大大降低了運營開銷。

Patel 說：“如果我們能構建一個本地 Kubernetes 叢集，我們就能將雲的力量帶到本地快速調配資源。之後對於開發人員來說，他們面向的接口是 Kubernetes；他們甚至可能沒有意識到或不關心他們的服務現在部署在內部，因爲他們只與 Kubernetes 一起工作。

因此，Patel 熱切關注 Kubernetes 多叢集能力的發展。他說：“有了叢集聯合，我們可以將內部部署作爲主叢集，將雲作爲輔助可突發叢集。因此，當有周年銷售或黑色星期五銷售並且我們需要更多的容器時，我們可以上雲。”

這種可能性以及 Grigoriu 和 Patel 的團隊已經使用Kubernetes所提供的影響，是 Nordstrom 最初在雲原生之旅中所起的作用。Grigoriu 說：“在當下的零售模式下，我們正在努力在力所能及的地方建立響應能力和靈活性。Kubernetes 使得爲開發端和運維端同時帶來效率的提升，這是一個雙贏。”
